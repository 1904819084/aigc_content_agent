# AIGC短视频Agent平台 Spec

## Why

需要设计一款面向短视频生产场景的 AIGC Agent 平台，把“创意生成、剧本生成、分镜规划、画面生成、镜头视频生成、素材混剪、成片导出”等链路整合为可配置、可追踪、可迭代的智能生产流程。当前目标不是直接编码，而是先形成一份可指导后续产品设计与工程实现的产品文档与技术文档。

## What Changes

- 设计一套 AIGC 短视频 Agent 平台的产品定义，明确目标用户、核心场景、关键流程、能力边界与版本规划
- 设计一套基于 Agent 架构的技术方案，覆盖多 Agent 分工、工作流编排、状态管理、模型接入、素材资产管理、可观测性与成本控制
- 定义从“短视频剧本生成”到“视频混剪导出”的标准化生产链路
- 定义系统输入输出、关键数据实体、任务状态流转与人工介入点
- 规划 MVP 范围与后续演进能力，避免首期方案过度设计

## Impact

- Affected specs: 产品能力定义、Agent 工作流编排、视频生产任务系统、素材资产管理、模型服务接入、运营监控能力
- Affected code: 暂无既有代码，预计后续影响 Web 控制台、工作流编排服务、模型适配层、素材存储服务、任务队列与渲染服务

## 产品文档

### 1. 产品定位

- 产品名称：AIGC 短视频 Agent 平台
- 产品目标：让内容团队或个人创作者以“一个主题或一句需求”为起点，自动完成从创意到成片的短视频生产
- 产品形态：Web 工作台 + Agent 编排引擎 + 多模型生成服务 + 任务渲染服务
- 产品价值：
  - 降低短视频制作门槛
  - 缩短从创意到成片的时间
  - 提升脚本、分镜和素材风格一致性
  - 支持人工审阅与局部重做，避免整链路返工

### 2. 目标用户

- 内容运营：批量生产营销、种草、资讯类短视频
- MCN/工作室：希望以标准化流程提升产能
- 独立创作者：想快速产出适配抖音、快手、视频号等平台的内容
- 品牌市场团队：希望围绕商品、活动、品牌主题快速生成视频素材

### 3. 核心使用场景

- 输入一个选题，自动生成多个剧本方向和镜头方案
- 输入商品卖点，自动生成带口播节奏的营销短视频
- 输入已有文案或文章，自动拆解成分镜并生成视频草稿
- 输入已有素材库，自动完成镜头匹配、混剪与字幕包装
- 对中间结果进行人工修改后继续后续流程，而不是重新开始

### 4. 核心流程

1. 用户输入主题、平台、时长、风格、受众、目标动作
2. 剧本 Agent 生成多个候选剧本
3. 分镜 Agent 将剧本拆解为镜头级分镜脚本
4. 画面生成 Agent 为每个镜头生成分镜图或关键帧图
5. 视频生成 Agent 基于分镜图、提示词和运动描述生成镜头视频
6. 混剪 Agent 结合素材库、配音、字幕、BGM、转场生成成片
7. 质检 Agent 检查时长、画幅、字幕、违禁词、镜头连贯性
8. 用户可对任意节点执行“重试、改写、人工编辑、局部重生成”
9. 导出成片及结构化生产记录

### 5. 核心功能模块

- 项目工作台：新建项目、查看任务状态、预览中间产物、导出成片
- 需求输入面板：主题、平台、时长、风格、商品信息、素材偏好、配音要求
- 剧本生成模块：生成多版本脚本，支持一键改写、扩写、缩写
- 分镜脚本生成模块：输出镜头描述、时长、景别、机位、人物动作、旁白、字幕建议
- 分镜图生成模块：生成镜头参考图，支持风格统一和角色一致性
- 分镜视频生成模块：按镜头生成视频片段，支持文生视频、图生视频、模板生成
- 视频混剪模块：拼接镜头、匹配素材、添加字幕、BGM、音效和包装元素
- 审阅与修订模块：对脚本、分镜、画面、视频片段逐层审阅
- 素材资产模块：管理图片、视频、音频、品牌模板、角色设定、风格预设
- 运行监控模块：查看任务日志、模型调用成本、失败原因、重试记录

### 6. MVP 范围

- 支持单项目单成片流程
- 支持 15 秒到 60 秒竖版短视频
- 支持文本输入生成剧本、分镜、分镜图、镜头视频、成片
- 支持有限人工编辑：剧本改写、镜头重生成、成片重新混剪
- 支持接入 1 类文本模型、1 类图像模型、1 类视频模型
- 支持任务状态追踪和失败重试

### 7. 非 MVP 能力

- 多人协作与权限系统
- 批量项目并行生产
- 自动发布到内容平台
- 精细化素材版权审核
- 高阶品牌数字人和多角色长期记忆

## 技术文档

### 1. 总体架构

- 前端工作台：负责项目创建、参数输入、结果预览、人工修订和导出操作
- API 网关层：负责鉴权、项目接口、任务提交、查询与中间结果读写
- Agent 编排层：基于工作流驱动多个 Agent 按阶段协作
- 模型适配层：统一封装文本、图像、视频、语音、字幕相关模型能力
- 任务执行层：负责任务队列、异步执行、重试、超时、并发控制
- 资产管理层：统一存储脚本、分镜、图片、视频片段、音频、字幕和最终成片
- 观测治理层：记录日志、指标、调用链、成本与失败原因

### 2. Agent 分工设计

- Orchestrator Agent：负责解析目标、组装工作流、调度下游 Agent、合并上下文
- Script Agent：生成选题分析、脚本大纲、分段文案、口播台词
- Storyboard Agent：把脚本拆解为镜头清单和镜头描述
- Image Prompt Agent：为每个镜头构造统一风格的绘图提示词
- Image Generation Agent：生成分镜图或关键帧参考图
- Video Prompt Agent：将镜头语义转成视频生成提示词和运动指令
- Video Generation Agent：生成镜头级视频片段
- Editing Agent：根据镜头顺序、素材可用性、字幕和音轨策略输出时间线
- QA Agent：检查内容安全、格式规范、连贯性与素材缺失问题
- Memory Agent：沉淀项目风格、角色设定、品牌约束、历史偏好

### 3. 工作流编排设计

- 工作流应采用有向图而不是单线串行，允许局部节点失败后仅重跑受影响节点
- 每个节点定义输入、输出、依赖关系、重试策略、幂等键和成本标签
- 编排引擎需支持以下能力：
  - 同步创建项目与参数快照
  - 异步投递长耗时生成任务
  - 节点级暂停、取消、恢复、重跑
  - 人工确认点插入，例如“脚本确认后再进入分镜”
  - 保存中间产物版本，支持回退

### 4. 核心数据实体

- Project：项目主实体，包含主题、平台、目标、状态、作者信息
- CreativeBrief：输入需求快照，记录用户最初参数
- ScriptVersion：剧本版本，记录文案、风格标签、生成参数、用户修改记录
- StoryboardVersion：分镜版本，记录镜头序号、时长、画面描述、台词、字幕、镜头运动
- ShotAsset：镜头资产，记录分镜图、视频片段、音频片段、封面图
- TimelineDraft：混剪草稿，记录镜头拼接顺序、字幕轨、BGM 轨、特效轨
- PublishVideo：导出成片，记录导出参数、分辨率、码率、导出时间
- Job：任务实体，记录执行阶段、状态、重试次数、耗时、失败原因、成本

### 5. 关键状态流转

- 项目状态：
  - `draft`：创建未启动
  - `script_generating`
  - `script_reviewing`
  - `storyboard_generating`
  - `asset_generating`
  - `editing`
  - `qa_reviewing`
  - `completed`
  - `failed`
- 节点状态：
  - `pending`
  - `running`
  - `waiting_user`
  - `succeeded`
  - `retryable_failed`
  - `fatal_failed`
  - `canceled`

### 6. 模型接入策略

- 文本模型：用于选题理解、剧本生成、分镜描述、字幕润色、质检解释
- 图像模型：用于分镜图生成、角色风格对齐、场景参考图生成
- 视频模型：用于镜头视频生成、图生视频、视频延展
- 语音模型：用于配音生成和口播多音色支持
- 音乐/音效模型或素材库：用于背景音乐和音效匹配
- 模型适配层必须统一以下接口：
  - 能力枚举
  - 请求参数标准化
  - 超时与重试
  - 成本统计
  - 审计日志
  - 回包结构标准化

### 7. 存储与资产管理

- 元数据建议使用关系型数据库存储项目、版本、任务和状态
- 中间产物与成片建议使用对象存储
- 大体积异步任务结果需记录可追踪 URI、生成时间、版本号和来源模型
- 资产管理需支持按项目和镜头维度检索，避免中间结果丢失

### 8. 前后端交互建议

- 创建项目接口：提交创作需求并返回项目 ID
- 启动工作流接口：按配置启动完整链路或指定节点链路
- 查询项目详情接口：返回项目状态和最近产物版本
- 查询节点结果接口：返回剧本、分镜、图片、视频等中间结果
- 节点重跑接口：允许按节点或镜头粒度局部重试
- 人工确认接口：对脚本、分镜、成片草稿进行确认后流转到下阶段

### 9. 可观测性与治理

- MVP 阶段先采用轻量观测方案，不引入 OpenTelemetry、Prometheus、Grafana
- 通过结构化日志记录节点耗时、任务状态、失败原因和模型调用摘要
- 通过数据库任务表保存项目级和节点级执行记录，便于排查生成质量问题
- 先提供基础成本统计、失败统计和简单限流能力
- 对内容安全增加审核节点，避免输出违规内容

### 10. 确认技术栈

- 前端工作台：React + TypeScript + Vite
- 前端状态管理：Zustand
- 前端样式方案：Less
- 前端媒体预览：原生 `video` 能力，必要时引入 Video.js
- 后端 API：Node.js + Express
- Agent 编排层：LangGraph.js
- 异步任务队列：BullMQ + Redis
- 关系型数据库：PostgreSQL
- 对象存储：S3 兼容对象存储
- 媒体处理：FFmpeg
- 实时状态通知：WebSocket
- 可观测性：结构化日志 + 任务执行记录表 + 基础管理后台统计

### 11. 技术栈确认说明

- 选择 Node.js + Express：统一前后端技术栈，保持服务层轻量灵活，适合快速搭建 API/BFF、任务编排入口和模型接入层
- 选择 LangGraph.js：更适合表达多 Agent 的状态图、条件分支、人工确认点和局部重跑能力
- 选择 BullMQ + Redis：适合承接图片生成、视频生成、混剪导出等长耗时异步任务，且与 Node.js 生态集成更顺畅
- 选择 PostgreSQL：适合管理项目、剧本版本、分镜版本、镜头资产、任务状态和审计日志
- 选择 S3 对象存储：适合保存分镜图、镜头视频、音频、字幕和最终成片等大文件资产
- 选择 FFmpeg：用于拼接、转码、抽帧、字幕压制、音视频混流，是视频混剪能力的基础
- 选择 React + Zustand + Less：适合构建复杂创作工作台与状态可视化页面，也贴合后续前端扩展需求
- 选择轻量观测方案：MVP 阶段先用结构化日志、任务表和基础统计页面满足排障与运营需要，后续再升级到完整 observability 体系

### 12. 分层落地建议

- 前端层：负责项目创建、参数填写、流程可视化、节点结果预览、人工确认和局部重跑
- API/BFF 层：负责鉴权、项目读写、任务触发、状态聚合和查询接口
- Agent 层：负责脚本、分镜、图像、视频、混剪、质检等领域能力
- Worker 层：负责模型调用、FFmpeg 执行、成片导出、资产上传和结果回写
- 存储层：PostgreSQL 存元数据，S3 存媒体资产，Redis 存队列与短期缓存

### 13. 风险与约束

- 视频生成成本高，必须控制并发与重试策略
- 多模型结果不稳定，需要版本化与人工确认机制兜底
- 镜头风格一致性是体验关键，需要角色和风格记忆能力
- 链路长且异步节点多，必须优先建设可回溯和可恢复能力
- MVP 阶段不建议一次覆盖多平台发布、多人协作、复杂版权审核

## ADDED Requirements

### Requirement: 提供端到端短视频生成流程

系统 SHALL 支持从主题输入到成片导出的端到端短视频生产流程，并在每个关键阶段产出结构化中间结果。

#### Scenario: 用户启动完整生产流程

- **WHEN** 用户输入主题、平台、时长和风格后启动生产
- **THEN** 系统按剧本、分镜、分镜图、分镜视频、混剪、质检的顺序或依赖关系执行任务
- **THEN** 用户可以查看每个阶段的中间结果和状态

### Requirement: 提供基于 Agent 的多阶段协作能力

系统 SHALL 使用多个职责清晰的 Agent 协同完成脚本、分镜、画面、视频和混剪任务，并由编排层统一管理依赖和状态。

#### Scenario: 多 Agent 协同执行

- **WHEN** Orchestrator Agent 收到项目任务
- **THEN** 它会按依赖顺序调度 Script Agent、Storyboard Agent、Image Generation Agent、Video Generation Agent、Editing Agent 和 QA Agent
- **THEN** 每个 Agent 的输入输出都被持久化记录

### Requirement: 支持局部重做与人工确认

系统 SHALL 允许用户在关键节点进行人工确认，并支持对单个节点或单个镜头执行局部重跑。

#### Scenario: 用户重做某个镜头

- **WHEN** 用户对某个分镜视频不满意并触发重做
- **THEN** 系统仅重跑该镜头及其后续受影响节点
- **THEN** 已确认且未受影响的节点结果应被保留

### Requirement: 支持任务状态追踪与失败恢复

系统 SHALL 记录任务状态、失败原因、重试记录和成本信息，以支持定位问题和恢复执行。

#### Scenario: 节点执行失败

- **WHEN** 某个模型调用超时或生成失败
- **THEN** 系统记录失败原因、上下文和重试次数
- **THEN** 对可恢复任务执行自动重试或允许用户手动重试

## MODIFIED Requirements

### Requirement: 产品首期范围控制

系统 SHALL 以 MVP 为目标优先覆盖文本到成片的单项目生产流程，不在首期纳入多人协作、自动发布和复杂版权治理能力。

#### Scenario: 首期能力范围评估

- **WHEN** 团队评估首期上线范围
- **THEN** 以单项目、单成片、可人工介入的完整生产闭环为交付边界

## REMOVED Requirements

### Requirement: 一次性覆盖全平台运营能力

**Reason**: 自动分发、平台运营、多账号管理与当前“先完成生产闭环”的目标不一致，会显著增加首期复杂度。
**Migration**: 在 MVP 验证完成后，以独立的运营分发能力增量扩展，不阻塞当前产品与技术设计落地。
